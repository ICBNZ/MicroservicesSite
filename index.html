Software Architecture

Microservices

Micro-services is a software architecture which is made up of independent services which serve a specific function. Microservices came about with the introduction of object oriented programming and service-oriented architecture where applications then began to be built as loosely coupled feature services as opposed to one large system as before with monolithic structures. Microservices further broke these service components into small specific tasks with independently built components able to be written in any programming language and connected to it’s own separate database, utilising APIs for communication and scalable cloud infrastructures.

Principles

Failure Isolation

As the services are independent of one another, Microservices are more resistant to failures. In the event a component fails the communication to that service can be cut, while the rest of the application continues to run. This is not the case with monolithic applications where one fault could stop the entire application running. Faults are also more easily identified within the small components and separate development teams.

Decentralised

Microservices uses different databases for the different services unlike that of monolithic systems which have a central database for all services. As each service uses its own separate database this allows them to be entirely different systems and allowing changes to be easily made to individual services.

Independently Deployable

Defining boundaries of the multiple components by encapsulation of each single function service’s details and own database so that services can be changed without affecting others.

Loose Coupling

Loose coupling involves systems where the components are separated into multiple small components with minimal dependencies. Small service components are each developed and operated independently of other services with minimal dependencies focusing on a single function, this allows the services to update and evolve without affecting or relying on the rest of the application and can be built with own programming languages.

Advantages of Micro-services



Programming Languages

Different programming languages and technology stacks can be used for each different service. This gives the development teams freedom to choose the best tools for that component and the independence of the services allows for the ability to easily re-write services and with no long-term commitment to technology stacks required, services can be easily updated with new technologies integrated.

Failure Resilence

Microservices are failure resistant, as the services are independent this allows for no single point of failure which prevents the whole application from crashing when one service is down.

Continuous Development/ Agility

Microservices with its small, independent services can provide continuous integration and delivery at a fast pace. Through continuous delivery with DevOps this allows services to quickly update with changes only required to be applied for that particular service, allowing the application to run as normal during the process.
The small size of the services means they are simpler and easier to understand which aids in training new team members. They are highly maintainable and testable and easier to deploy with developers able to easily experiment and develop new features, shortening development processes to just weeks.

Low Costs/ Scaling

Change can occur at a high rate due to small independent services with faster iterations, continuous development and integration without affecting the entire application.
Microservices also allows each service to individually scale according to their requirements and demand for the feature, which allows for accurate infrastructure needs and measurements of cost and ability to be available when demand increases, which both reduce downtime.
With quicker to market features, more efficient code, infrastructure and cloud-based servers, costs are reduced for the business. Also, there can be increased revenue as user engagement increases with the application able to readily implement new features and improve.

Disadvantages of Microservices

  • Some disadvantages of Microservices include duplication of data that may occur as there are separate databases for the different services.
  • Some services also aren’t able to be easily split into the different separate components.

Microservices vs Service Oriented Architecture (SOA)

Similarities

Languages

Both Microservices and Service Oriented architecture allow the use multiple programming languages to be used for the different services within an application. However, with SOA the development teams need to share communication mechanisms whereas MS are able to be independently deployed.

Services

Both architectures are focused on services as the main component, divided into service modules which are "black boxes" to users, as opposed to Monolithic systems.

Distrubuted System

Both architectures have distributed systems where a means of communication is needed, however the communication differs between the two.

Differences

Size

SOA has coarse grained feature components which can be subsystems, whereas microservices is broken down into smaller, fine grained single-task focused components.

Communication

Microservices focuses on communicating and co-ordinating as little as possible with the other teams and components whereas SOA is the opposite. Communication with microservices is through light-weight API protocols and with SOA through middleware messaging where the components need to know details about the other services. Communication with SOA happens through the Enterprise Services Bus which means if one service slows down this can slow down the entire system, whereas microservices the services are independent, and failures only affect that particular service.

Decentralised

Microservices is focused on small independent, autonomous services with minimal dependencies. Microservices focuses on bounded contexts of single units and data independant from other services and SOA is focused on component sharing. Microservices has its own database for each service, whereas SOA have a shared database, and centralised communication.

Failure Isolation

As the Service Orientated Architecture communicates through the ESB, this can be a single point of failure which can affect the application, however with Microservices, there is no single point of failure as all components are separate and independant, so one service's failure doesn't affect the rest of the system.

Flexiblity

SOA have common standards and governance between components whereas Microservices offer more flexibility and freedom for teams, focusing on people and collaboration. The independance of the microservice services allows for easier updating, changing and faster deployment of services without affecting the rest of the application.

Real World Examples

Click on the logos below to find out how these well known companies use microservices

Amazon

Amazon started as a monolithic application however moved to microservices as it expanded to different areas and required a larger number of developers to maintain and build it. With the monolithic system, merging of changes took time and resulted in delays. With the microservices architecture, independent functions, decoupled services and cloud based scalable servers this meant Amazon could keep up with changing demands without wasting money and speed up development.

Uber

Ride sharing app Uber began as a tightly coupled, monolithic architecture system which contained all systems and processes for the business. However, as the business grew there were issues with scaling and development of new features which is why Uber moved to microservices which allows teams to quickly develop and deploy product features using the best updated tools.

Netflix

Netflix is one of the first companies to use Microservices when it began moving to cloud based microservices in 2009 initially with non-customer based services such as movie encoding and then all services finally moved over, at this point the term microservices didn't exist. Reasons behind the move to the cloud based microservices were the fact Netflix needed to be fast, increasing demand and the need for services to be available all the time. In 2008 a missing semicolon led to database corruption and it was down for hours this led to Netflix deciding to change their system. Netflix now has an API gateway which handles around 2 billion requests per day with developers able to deploy code thousands of times a day.

eBay

eBay uses microservices with more than 1000 services and development teams for each. With front end APIs and backend microservices handling tasks such as shipping. Microservices allows eBay development teams to easily update and deploy new functions with their cloud portal.
